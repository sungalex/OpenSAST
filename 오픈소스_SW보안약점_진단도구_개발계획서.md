# 오픈소스 기반 SW 보안약점 진단도구 개발 계획서

**"행정기관 및 공공기관 정보시스템 구축·운영 지침" 구현단계 49개 보안약점 점검 기준 충족**

---

## 1. 프로젝트 개요

### 1.1 배경 및 필요성

현행 상용 SW 보안약점 진단도구는 다음과 같은 한계를 갖고 있다:

- **개발환경 추종 부족**: 신규 언어(Go, Rust, Kotlin 등), 최신 프레임워크(Spring Boot 3.x, Next.js 14+, FastAPI 등), 컨테이너/클라우드 네이티브 환경에 대한 대응이 느림
- **높은 오탐률**: 정적분석 특성상 컨텍스트를 충분히 반영하지 못해 오탐(False Positive)이 전체 탐지의 30~60%에 달하는 경우가 빈번
- **제한된 언어 지원**: 공인 진단도구 대부분이 Java, C, C++ 중심이며 Python, JavaScript/TypeScript, Go, Kotlin, Swift 등 현대적 언어 지원이 미흡
- **높은 도입 비용**: 라이선스 비용이 수천만 원~수억 원 수준으로, 소규모 기관이나 사업에서 도입이 어려움
- **폐쇄적 룰 체계**: 커스텀 룰 작성이 어렵거나 불가능하여, 기관 고유 보안 정책 반영이 제한적

### 1.2 목표

오픈소스 SAST 엔진을 핵심으로 활용하여, **행안부 소프트웨어 보안약점 진단가이드(2021) 구현단계 49개 항목**을 완전히 커버하는 통합 진단도구를 개발한다.

### 1.3 핵심 설계 원칙

| 원칙 | 설명 |
|------|------|
| **다중 엔진 오케스트레이션** | 단일 도구의 한계를 극복하기 위해 복수의 오픈소스 엔진을 조합하여 탐지율 극대화 및 오탐 최소화 |
| **CWE 기반 룰 매핑** | 49개 보안약점 항목을 CWE ID로 정밀 매핑하고, 각 CWE별 최적 엔진을 지정 |
| **LLM 기반 오탐 필터링** | AI를 활용하여 탐지 결과를 자동 분류하고, 오탐 확률을 산출하여 진단원의 검증 부담 경감 |
| **확장 가능한 룰 체계** | YAML 기반 커스텀 룰을 쉽게 작성·배포할 수 있는 구조 |
| **개발 파이프라인 통합** | CI/CD, IDE, Git Hook 등 개발 전 과정에 자연스럽게 통합 |

## 2. 법적·제도적 점검 기준 분석

### 2.1 관련 법령 및 지침

- **행정기관 및 공공기관 정보시스템 구축·운영 지침** (행안부고시 제2025-1호, 2025.1.2. 시행)
  - 제51조: 소프트웨어 개발보안 적용
  - 제52조: 설계단계 보안설계 기준 (20개 항목)
  - 제53조: 구현단계 보안약점 제거 기준 (49개 항목)
- **소프트웨어 보안약점 진단가이드** (2021.11. 개정, KISA)
- **정보시스템 감리기준** 제10조 제1항: 세부검사항목 포함 의무

### 2.2 구현단계 보안약점 49개 항목 분류

| 분류 | 항목 수 | 주요 보안약점 |
|------|---------|--------------|
| **입력데이터 검증 및 표현** | 18개 | SQL 삽입, XSS, 경로조작, OS 명령어 삽입, 코드 삽입, SSRF, 위험한 형식 파일 업로드, 신뢰되지 않는 URL 주소로 자동접속 연결, XQuery 삽입, XPath 삽입, LDAP 삽입, HTTP 응답분할, 정수형 오버플로우, 보안기능 결정에 사용되는 부적절한 입력값, 메모리 버퍼 오버플로우, 포맷 스트링 삽입, 서버사이드 요청위조(SSRF), 신뢰할 수 없는 데이터의 역직렬화 |
| **보안기능** | 12개 | 적절한 인증 없는 중요기능 허용, 부적절한 인가, 중요자원에 대한 잘못된 권한설정, 취약한 암호화 알고리즘 사용, 암호화되지 않은 중요정보, 하드코드된 중요정보, 충분하지 않은 키 길이 사용, 적절하지 않은 난수값 사용, 취약한 비밀번호 허용, 부적절한 전자서명 확인, 부적절한 인증서 유효성 검증, 사용자 하드디스크에 저장되는 쿠키를 통한 정보노출 |
| **시간 및 상태** | 2개 | 검사시점과 사용시점(TOCTOU), 종료되지 않는 반복문 또는 재귀함수 |
| **에러처리** | 3개 | 오류메시지를 통한 정보노출, 오류 상황 대응 부재, 부적절한 예외처리 |
| **코드오류** | 7개 | Null Pointer 역참조, 부적절한 자원 해제, 해제된 자원 사용, 초기화되지 않은 변수 사용, 신뢰할 수 없는 데이터의 역직렬화, 잘못된 세션에 의한 데이터 정보노출, 제거되지 않고 남은 디버그 코드 |
| **캡슐화** | 5개 | 잘못된 세션에 의한 데이터 정보노출, 제거되지 않고 남은 디버그 코드, 시스템 데이터 정보노출, Public 메소드부터 반환된 Private 배열, Private 배열에 Public 데이터 할당 |
| **API 오용** | 2개 | DNS Lookup에 의존한 보안결정, 취약한 API 사용 |

### 2.3 진단도구 인증 관련 사항

현행 지침은 **"KISA 연구소장이 인증한 보안약점 진단도구"** 사용을 의무화하고 있다. 따라서 본 도구의 최종 목표는 KISA CC(공통평가기준) 인증 획득이며, 개발 과정에서 인증 요구사항을 선제적으로 반영한다.

> **참고**: 인증 획득 전에도 "진단원에 의한 수동 진단"과 병행하여 활용 가능하며, 오픈소스 도구를 보조 도구로 사용하는 것은 지침상 제한되지 않음

### 6.2 CodeQL 쿼리 예시: TOCTOU (CWE-367, Java)

```ql
/**
 * @name 검사시점과 사용시점 경쟁조건 (TOCTOU)
 * @description 파일 존재 여부 확인 후 사용 사이에 경쟁조건 발생 가능
 * @kind path-problem
 * @id mois/toctou-race-condition
 * @problem.severity warning
 * @tags security cwe-367 mois-sr3-1
 */

import java
import semmle.code.java.dataflow.TaintTracking

class FileExistsCheck extends MethodAccess {
  FileExistsCheck() {
    this.getMethod().hasName("exists") and
    this.getMethod().getDeclaringType().hasQualifiedName("java.io", "File")
  }
}

class FileUse extends MethodAccess {
  FileUse() {
    this.getMethod().getDeclaringType().hasQualifiedName("java.io", "File") and
    this.getMethod().hasName(["delete", "renameTo", "createNewFile",
                              "mkdir", "mkdirs", "setReadable", "setWritable"])
  }
}

from FileExistsCheck check, FileUse use
where
  check.getQualifier().getType() = use.getQualifier().getType() and
  check.getEnclosingStmt().getParent*() = use.getEnclosingStmt().getParent*()
select use, check, use,
  "파일 존재 확인(" + check.getLocation().getStartLine() + "행) 후 " +
  "파일 조작(" + use.getLocation().getStartLine() + "행) 사이에 TOCTOU 경쟁조건 가능"
```
